NeuralBridge Product Overview
Introduction

What is NeuralBridge?

NeuralBridge is an AI-native Android automation platform that enables AI agents — Claude Code, Cursor, Copilot, or any MCP-compatible client — to control Android devices with the same speed and fluidity as a human hand. Every design decision flows from a single constraint:

Core Constraint

An AI agent must be able to do everything a human finger and eye can do, in under 100ms, with zero human intervention.

The Problem

Why Existing Tools Fall Short

Every existing Android automation solution routes through ADB for actions, meaning each command is a separate process spawn plus a USB/TCP round-trip. This adds 500–2,000ms of unavoidable overhead — fatal for real-time AI agent loops.

❌ Old Approach (Appium / UIAutomator2)
WebDriver → HTTP → UIAutomator2 → IPC → ADB

Each layer adds latency. Polling required. No event streaming. 200–2,000ms per action.
✅ NeuralBridge Approach
MCP → TCP → AccessibilityService (in-process)

Persistent connection. Event-driven. In-process execution. <100ms per action.

NeuralBridge Architecture
System Design

Three-Tier Architecture

NeuralBridge uses a three-tier design separating AI agent communication, host-side protocol handling, and on-device execution into distinct layers with well-defined interfaces.

Fast Path <100ms

Gestures, UI observation, screenshots, text input — executed in-process via AccessibilityService. Zero IPC overhead.

Slow Path 200–500ms

App install, clear data, force-stop, grant permissions, WiFi control — routed through MCP server's ADB connection.

Event Streaming

Device pushes UI changes, notifications, toasts, crashes via the same TCP connection using Event envelope (type 0x03).

NeuralBridge Connection & Action Flow
How It Works

End-to-End Action Flow

The complete sequence when an AI agent calls android_tap(text="Login") — from tool invocation to gesture execution on device. Total round-trip: ~90ms.

Action Sequence

1
AI Agent → MCP Server
<1ms
Tool call via stdio pipe: android_tap(text="Login"). MCP JSON-RPC message deserialized by rmcp server.
2
Semantic Resolver
<1ms (cache hit) / <50ms (miss)
Checks cached UI tree for element with text="Login". Resolves to center coordinates, e.g. (240, 850). Fuzzy match fallback if exact fails.
3
MCP Server → Companion App
<5ms
Encodes TapRequest{coordinates: Point(240,850)} as Protobuf. Sends over TCP socket (localhost port-forwarded via ADB).
4
Gesture Execution
<50ms
AccessibilityService builds GestureDescription with 50ms single-stroke at (240,850). dispatchGesture() injects real touch events.
5
UI Change Event (pushed)
<30ms
AccessibilityService fires TYPE_VIEW_CLICKED event. New UI tree delta computed and pushed as Event message to MCP server.
6
Response to AI Agent
<1ms
MCP server returns {success: true} via stdio. Agent proceeds with next action. Total: ~90ms.

Why NeuralBridge Wins

Head-to-Head Comparison

The fundamental advantage: NeuralBridge keeps a persistent in-process connection to a service that can already see and touch everything. Every other tool spawns a new process or sends a new ADB command per action.

Tool Approach Gesture Latency AI-Native Protocol Event-Driven Root Required Speedup
NeuralBridge MCP → TCP → AccessibilityService (in-process) <100ms ✓ MCP ✓ ✗ Baseline
Appium WebDriver → HTTP → UIAutomator2 → IPC → ADB 200–500ms ✗ WebDriver ✗ Polling ✗ 10–20× slower
mobile-mcp ADB shell commands, polling 500–2,000ms ~ MCP (partial) ✗ Polling ✗ 5–10× slower
Android-MCP Python + UIAutomator2 + ADB 200–1,000ms ~ MCP (partial) ✗ ✗ 5–15× slower
UIAutomator2 (direct) Separate process, IPC 200–500ms ✗ REST ✗ ✗ 10–20× slower
ADB Shell (raw) USB/TCP + process spawn per command 500–2,000ms ✗ Shell ✗ Polling ✗ >25× slower†

† ADB Shell latency of 500–2,000ms vs NeuralBridge ~20ms yields a practical range of 25×–100×. The 10–20× speedup vs Appium and 5–15× vs Android-MCP are sourced directly from the architecture specification (PRD section 6).

Why Privileged Operations Are Slower — By Design

Privileged operations are intentionally routed through the MCP server's ADB connection rather than attempted from the companion app (where they would fail with "Permission denied"). This trades latency for reliability and security. The companion app runs as a regular user-space process and cannot execute privileged shell commands.

Distribution & Installation

⚠️ Sideload Only — No Google Play

Google Play policy prohibits AccessibilityService apps that primarily automate rather than assist users with disabilities. NeuralBridge must be distributed as a direct APK via ADB.

adb install neuralbridge-companion.apk

Two Permissions Required

Full functionality requires two separate user permission grants:

1. AccessibilityService — Settings → Accessibility → NeuralBridge
2. NotificationListenerService — Settings → Notifications → Notification access → NeuralBridge

Semantic Element Resolver — Resolution Priority

When an agent specifies a selector, resolution proceeds through 6 levels until a match is found:

#Match TypeExampleNotes
1Exact text matchtext="Login"Case-sensitive exact match on element.text
2Partial text matchtext="Log in"element.text.contains(selector.text)
3Content descriptioncontent_desc="Login button"element.contentDescription == selector.description
4Resource ID suffixresource_id="login_button"element.resourceId.endsWith(selector.resourceId)
5Combined matchtext="Login" + resource_id="btn"AND logic — all provided selector fields must match
6Fuzzy matchtext="Logon"Levenshtein distance < 3 (fallback)

If multiple elements match: prefer visible → prefer interactive (clickable) → prefer closest to screen center. If still ambiguous, returns ELEMENT_AMBIGUOUS with all candidates.

Keep-Alive Strategy

Android aggressively kills background services. NeuralBridge prevents this with: (1) Foreground Service with persistent notification, (2) Battery optimisation exemption request, (3) Watchdog coroutine restarts TCP server if killed, (4) Partial wake lock only during active automation sessions.

Gesture Limits (dispatchGesture)

Per Android API documentation:

• Max simultaneous touch points: 10
• Max gesture duration: 60 seconds
• Strokes can overlap in time (true multi-touch)
• Tap stroke duration: 50ms
• Long press stroke duration: ≥1,000ms
